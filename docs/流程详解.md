# Mem0 实验流程详解

## 📋 三个命令的完整流程

### 命令 1: `python run_experiments.py --technique_type mem0 --method add`

**目的**: 将对话数据提取并存储为记忆（Memories）

**详细流程**:

```
1. 读取数据集
   └─> 加载 dataset/locomo10.json
   └─> 包含多个对话会话（conversations）

2. 对每个对话会话:
   ├─> 提取两个说话者（speaker_a 和 speaker_b）
   ├─> 为每个说话者创建唯一的 user_id（格式：speaker_name_idx）
   ├─> 删除该用户之前的所有记忆（清空）
   │
   └─> 处理对话中的每个时间段的对话:
       ├─> 将对话转换为消息格式
       │   ├─> speaker_a 的消息 → role: "user"
       │   └─> speaker_b 的消息 → role: "assistant"
       │
       ├─> 调用 Mem0 API 的 add() 方法
       │   ├─> 使用 LLM 提取记忆（根据 custom_instructions）
       │   ├─> 将记忆转换为向量（embedding）
       │   └─> 存储到向量数据库（Vector Store）
       │
       └─> 同时为两个说话者添加记忆（多线程并行）
           ├─> speaker_a 看到自己的消息作为 user
           └─> speaker_b 看到自己的消息作为 user（角色反转）
```

**关键点**:
- **记忆提取**: Mem0 使用 LLM（根据 `custom_instructions`）从对话中提取结构化记忆
- **向量化**: 每个记忆被转换为 embedding 向量（使用 embedding 模型，默认 `text-embedding-3-small`）
- **存储**: 记忆存储在向量数据库中，支持语义搜索

**输出**: 记忆存储在 Mem0 平台/数据库中，每个用户有独立的记忆空间

### 命令 2: `python run_experiments.py --technique_type mem0 --method search --top_k 30 --output_folder results/`

**目的**: 对每个问题检索相关记忆并生成回答

**详细流程**:

```
1. 读取数据集
   └─> 加载 dataset/locomo10.json
   └─> 包含问题和答案（QA pairs）

2. 对每个对话会话:
   ├─> 获取说话者 ID（speaker_a_idx, speaker_b_idx）
   │
   └─> 对每个问题:
       │
       ├─> 步骤 1: 记忆检索（为两个说话者分别检索）
       │   ├─> 对 speaker_1: search_memory(user_id, question)
       │   │   └─> Mem0 内部流程:
       │   │       ├─> 1. 将问题转换为 embedding 向量
       │   │       ├─> 2. 在向量数据库中搜索相似记忆
       │   │       │   └─> 使用余弦相似度（cosine similarity）
       │   │       ├─> 3. 应用过滤器（user_id, filters）
       │   │       ├─> 4. 返回 top_k=30 个最相关的记忆
       │   │       └─> 5. 每个记忆包含：内容、时间戳、相似度分数
       │   │
       │   └─> 对 speaker_2: search_memory(user_id, question)
       │       └─> 同样的流程
       │
       ├─> 步骤 2: 构建提示词（Prompt）
       │   └─> 使用 ANSWER_PROMPT 模板
       │       ├─> 包含 speaker_1 的 top_k 个记忆
       │       ├─> 包含 speaker_2 的 top_k 个记忆
       │       ├─> 包含问题
       │       └─> 要求 LLM 基于记忆回答问题（少于5-6个词）
       │
       ├─> 步骤 3: 生成回答
       │   └─> 调用 LLM（GPT-4o-mini 或 DeepSeek）
       │       └─> 输入: 提示词
       │       └─> 输出: 生成的回答
       │
       └─> 步骤 4: 保存结果
           └─> 保存到 results/mem0_results_top_30_filter_False_graph_False.json
               ├─> 问题
               ├─> 正确答案
               ├─> 生成的回答
               ├─> 检索到的记忆（speaker_1_memories, speaker_2_memories）
               ├─> 记忆数量
               └─> 检索和生成的时间
```

**关键点**:
- **向量检索**: 使用语义相似度搜索，不是关键词匹配
- **top_k=30**: 返回最相关的 30 个记忆
- **双说话者检索**: 为两个说话者分别检索，因为问题可能涉及双方的信息

**输出**: `results/mem0_results_top_30_filter_False_graph_False.json`

### 命令 3: `python evals.py --input_file results/mem0_results_top_30_filter_False_graph_False.json --output_file evaluation_metrics.json --max_workers 2`

**目的**: 评估生成回答的质量

**详细流程**:

```
1. 读取结果文件
   └─> 加载 mem0_results_top_30_filter_False_graph_False.json

2. 对每个问题-回答对（多线程并行，max_workers=2）:
   │
   ├─> 计算 BLEU 分数
   │   └─> 比较生成回答和正确答案的 n-gram 重叠度
   │       ├─> BLEU-1: 单词级别
   │       ├─> BLEU-2: 双词组合
   │       ├─> BLEU-3: 三词组合
   │       └─> BLEU-4: 四词组合
   │
   ├─> 计算 F1 分数
   │   └─> 基于单词集合的精确率和召回率
   │       ├─> 精确率 = 共同单词数 / 生成回答单词数
   │       ├─> 召回率 = 共同单词数 / 正确答案单词数
   │       └─> F1 = 2 * (精确率 * 召回率) / (精确率 + 召回率)
   │
   └─> LLM Judge 评分
       └─> 使用 LLM（DeepSeek 或 GPT-4）判断回答是否正确
           ├─> 输入: 问题、正确答案、生成回答
           ├─> LLM 分析语义是否匹配
           └─> 输出: CORRECT (1) 或 WRONG (0)
           │
           └─> LLM Judge 的评分标准:
               ├─> 语义匹配即可（不要求完全一致）
               ├─> 对于时间问题，相对时间转换为绝对时间也算正确
               ├─> 格式不同但内容相同也算正确
               └─> 比较宽松的评分标准

3. 保存评估结果
   └─> 保存到 evaluation_metrics.json
       ├─> 每个问题的 BLEU、F1、LLM 分数
       └─> 按会话组织
```

**关键点**:
- **BLEU**: 衡量文本相似度（基于 n-gram）
- **F1**: 衡量单词级别的匹配度
- **LLM Judge**: 语义级别的判断，更接近人类评估

**输出**: `evaluation_metrics.json`

## 🔍 检索机制详解

### 使用的工具和技术

Mem0 使用 **向量数据库（Vector Database）** 进行语义检索：

1. **Embedding 模型**
   - 默认: `text-embedding-3-small` (OpenAI)
   - 作用: 将文本转换为高维向量（embedding）
   - 特点: 语义相似的文本会有相似的向量

2. **向量搜索**
   - 方法: **余弦相似度（Cosine Similarity）**
   - 原理: 计算查询向量和记忆向量的夹角
   - 公式: `similarity = cos(θ) = (A·B) / (||A|| * ||B||)`
   - 范围: -1 到 1，值越大越相似

3. **检索流程**
   ```
   问题文本
      ↓
   Embedding 模型
      ↓
   问题向量 (query vector)
      ↓
   向量数据库搜索
      ├─> 计算与所有记忆的相似度
      ├─> 按相似度排序
      ├─> 应用过滤器（user_id 等）
      └─> 返回 top_k 个最相似的记忆
   ```

4. **向量数据库**
   - Mem0 Platform: 使用托管的向量数据库
   - 支持多种后端: MongoDB, Pinecone, Qdrant 等
   - 特点: 支持大规模向量搜索，快速检索

### 为什么使用向量检索？

- **语义理解**: 不是简单的关键词匹配
  - 例如: "什么时候丢了工作" 和 "失业的时间" 会被识别为相似
- **上下文理解**: 理解问题的意图，而不仅仅是字面意思
- **多语言支持**: 向量空间可以跨语言工作

## 📊 top_k 参数详解

### 什么是 top_k？

**top_k = 30** 表示：返回与问题最相关的 **30 个记忆**

### top_k 的工作流程

```
1. 向量搜索计算所有记忆的相似度分数
   └─> 例如: 
       - 记忆1: 0.95 (非常相关)
       - 记忆2: 0.87
       - 记忆3: 0.82
       - ...
       - 记忆100: 0.12 (不太相关)

2. 按相似度分数从高到低排序

3. 取前 top_k 个
   └─> top_k=30 → 取前30个最相关的记忆

4. 返回给 LLM 用于生成回答
```

### top_k 是越大越好吗？

**答案：不一定！** 需要平衡多个因素：

#### ✅ top_k 较大的优点：

1. **更全面的信息**
   - 可能包含更多相关记忆
   - 减少遗漏重要信息的风险

2. **更好的上下文**
   - LLM 有更多信息来生成回答
   - 对于复杂问题（multi-hop）有帮助

#### ❌ top_k 较大的缺点：

1. **噪声增加**
   - 包含不相关的记忆
   - 可能干扰 LLM 的判断

2. **成本增加**
   - 更多的 tokens 输入到 LLM
   - API 调用成本更高

3. **延迟增加**
   - 处理更多记忆需要更多时间
   - 响应变慢

4. **上下文窗口限制**
   - LLM 有最大 token 限制
   - 太多记忆可能超出限制

### 如何选择 top_k？

**经验法则**:

1. **从较小值开始** (如 10-20)
   - 如果准确率不够，逐步增加
   - 观察准确率变化

2. **根据问题类型调整**
   - **简单问题** (single-hop): top_k=10-20 可能足够
   - **复杂问题** (multi-hop): top_k=30-50 可能更好
   - **开放域问题**: top_k=50+ 可能更好

3. **根据记忆数量调整**
   - 如果用户只有 50 个记忆，top_k=30 已经很多了
   - 如果用户有 1000+ 个记忆，top_k=30 可能不够

4. **实验验证**
   ```bash
   # 尝试不同的 top_k 值
   python run_experiments.py --technique_type mem0 --method search --top_k 10
   python run_experiments.py --technique_type mem0 --method search --top_k 30
   python run_experiments.py --technique_type mem0 --method search --top_k 50
   
   # 比较结果
   python evals.py --input_file results/... --output_file ...
   python generate_scores.py
   ```

### 论文中的 top_k

根据 Mem0 论文，通常使用：
- **基础 Mem0**: top_k=30
- **Mem0+ (graph-based)**: top_k=30（但会结合图结构）

### 实际建议

对于你的情况（当前 LLM Score = 41.62%，目标是 66.9%）：

1. **先检查模型配置**（最重要）
   - 确保使用 GPT-4o-mini 而不是 DeepSeek

2. **如果模型正确，尝试增加 top_k**
   ```bash
   # 尝试 top_k=50
   python run_experiments.py --technique_type mem0 --method search --top_k 50 --output_folder results/
   ```

3. **观察结果**
   - 如果准确率提升 → 继续优化
   - 如果准确率下降 → 可能噪声太多，尝试其他方法

## 💡 关键要点总结

1. **记忆提取**: 使用 LLM 从对话中提取结构化记忆
2. **向量检索**: 使用语义相似度搜索，不是关键词匹配
3. **top_k**: 返回最相关的 k 个记忆，需要平衡准确率和成本
4. **评估**: 使用 BLEU、F1 和 LLM Judge 三种指标
5. **问题**: 当前可能因为模型配置（DeepSeek vs GPT-4o-mini）导致分数偏低
